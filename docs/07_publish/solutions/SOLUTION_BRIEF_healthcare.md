# 3e-Aria-Gatekeeper for Healthcare

The hospital that adds an AI assistant to a clinical workflow makes a different kind of decision than the bank that adds one to a contact centre. The data the assistant sees — patient records, symptom descriptions, free-text notes from clinical encounters, lab results, medication lists — is some of the most sensitive personal data any organisation handles, and the consequences of getting access controls wrong are not just regulatory fines but potential patient harm. The clinician using the assistant has a different relationship with the information than the support agent at the bank: the clinician *needs* to see the patient identifiers, but the model the assistant calls out to does not.

This brief explains how 3e-Aria-Gatekeeper addresses AI governance in a healthcare context, with attention to the data minimisation discipline that healthcare organisations operate under and an honest accounting of what the v0.1.1 release does and does not do.

## What healthcare reads differently

Three things are distinctively true in healthcare AI deployments.

The data minimisation requirement is not a regulatory checkbox; it is structural. A clinical assistant that helps a physician summarise a patient encounter does not need the patient's name to be useful — it needs the symptoms, the relevant history, the medication list, and the diagnostic question. Sending the patient's name and date of birth to the LLM provider adds no clinical value and creates a much larger blast radius if the data is ever mishandled by the provider. The same is true for an administrative assistant that helps clinic staff draft a letter to a referring physician — the letter needs the relevant clinical context but not the patient's home address. Data minimisation in healthcare is not "send less data because regulation says so"; it is "send less data because there is no clinical reason to send more".

The role-based access pattern is more nuanced than most other industries. A clinician seeing a complete patient record is appropriate; the same clinician seeing patient data for an unrelated patient (because the assistant cross-referenced something) is not. An administrative user seeing PII-redacted patient summaries is appropriate; the same user seeing full medical records is not. A research user seeing fully de-identified data sets is appropriate; the same user seeing identifiable data is a HIPAA-equivalent violation in most jurisdictions. The access-control model is not binary; it has multiple roles with multiple data scopes, and the AI traffic between applications and LLM providers needs to respect the same model that the rest of the clinical-information-system access controls respect.

The audit requirement is genuinely strict. Healthcare audit frameworks (HIPAA in the US, KVKK + Personal Data Health Records regulations in Turkey, GDPR + national health-data laws in EU members, HDS in France, equivalent frameworks in MENA and APAC markets) typically require multi-year retention of access logs for any system that handles personal health information. A governance product that does not produce a durable, queryable, tamper-proof audit trail for AI traffic is producing an audit gap that the healthcare organisation will eventually have to explain.

## What Gatekeeper does for a healthcare AI integration

The three Gatekeeper modules each address a healthcare-specific concern.

**`aria-shield` provides the cost discipline and provider-routing that healthcare AI workloads need.** Per-team token quotas (clinical-AI team, administrative-AI team, research-AI team) and per-model dollar budgets create the per-function attribution that healthcare organisations already do for other infrastructure spending. Multi-provider routing means the organisation can switch between providers based on policy — for example, routing the most sensitive clinical workloads to a regionally-hosted, BAA-compliant Azure OpenAI endpoint while routing administrative workloads to a cheaper general-purpose provider. The circuit breaker means an outage at one provider does not take down the AI-assisted workflows that have been adopted; the threshold alerts mean budget conversations happen in time to do something about them.

**`aria-mask` provides the data-minimisation discipline that healthcare data egress requires.** The eight built-in PII patterns include several that matter directly for healthcare: TC Kimlik (Turkish national ID, with mod-11 checksum) for Turkish-jurisdiction deployments, MSISDN for patient phone numbers, IBAN for billing flows, email for patient communication addresses, and date-of-birth as a structural pattern. The role-based policy resolution distinguishes the clinical role (which sees most of the data, with selective masking for fields the clinician does not need for the current task) from the administrative role (which sees less) from the external-partner role (which sees the minimum required for the specific transaction). The twelve mask strategies include `redact` (replace with placeholder for fields the assistant does not need), `last4` (for verification flows where the user reads the last few digits back to confirm identity), `mask:dob` (standard date-of-birth masking), `mask:phone` (phone-number masking), and `mask:email` (email masking) — covering most of the field-level transformations a healthcare workflow would naturally configure.

For free-text clinical content where structural patterns do not apply — clinical notes, symptom descriptions, encounter summaries — the NER pipeline runs Apache OpenNLP for English and the multilingual Turkish-BERT model by default for Turkish content. The pluggable engine architecture means healthcare organisations can configure additional engines for clinical-specific NER (for example, a model trained specifically on clinical entity recognition that distinguishes medications, conditions, and procedures from generic person/location/organisation entities). The `NerEngine` interface is small enough that adding a clinical-NER engine is a matter of writing a wrapper around an existing model — most clinical NER models are available through HuggingFace and ONNX-exportable.

**`aria-canary` provides the change-safety discipline that healthcare infrastructure leaders already require.** Model upgrades in clinical workflows are particularly high-stakes — a regression in a clinical-decision-support assistant is a patient-safety event, not a metrics blip. Multi-stage progressive splitting with consistent-hashed routing means clinicians using the assistant during the rollout see consistent behaviour through the experiment. Auto-rollback on error-rate sustain provides the safety net for workflows where humans are not constantly checking the metrics dashboard. Traffic shadowing with structural diff between baseline and candidate model responses lets clinical-informatics teams see exactly how a candidate model differs from production on actual workload, before any clinical user sees the candidate's output. This is the kind of pre-deployment safety check that clinical change-control boards are already comfortable applying to other clinical-information-system changes.

## Compliance posture in healthcare contexts

Throughout this brief — and throughout all our documentation — we describe what Gatekeeper *does* in support of operator audits. We do not certify compliance with any framework. Healthcare organisations operate under audit boundaries that require many controls; Gatekeeper is one of them.

The relevant capabilities by framework:

For **HIPAA-equivalent regulations** (HIPAA itself in the US, KVKK + Personal Data Health Records regulations in Turkey, GDPR + national health-data laws in EU members), Gatekeeper supports data minimisation by masking PII before it reaches third-party LLM providers, role-based access through the role-policy resolution, and the durable seven-year audit trail (configurable for longer retention) that supports the multi-year audit requirements common across all these frameworks. **Gatekeeper is not a HIPAA-compliant product** — HIPAA compliance applies to covered entities and business associates and requires controls across the entire system, not just the gateway. Healthcare organisations should describe Gatekeeper as a control that supports the data-minimisation and access-logging requirements of their compliance posture, not as a control that certifies compliance.

For **GDPR / KVKK / PDPL** in healthcare contexts specifically, the additional consideration is that personal health data is a special category of personal data with stricter handling requirements. The PII masking applies the same way; the role-based policies map directly onto the consent and access-purpose model that healthcare organisations already implement; the audit trail provides the access-logging that data-protection regulators require for special-category data.

For **data sovereignty requirements**, the multi-provider routing means the organisation can route healthcare workloads to providers that have BAA / equivalent agreements and that operate in the appropriate jurisdiction, while routing other AI workloads to providers chosen for other reasons. The geographic-policy enforcement through consumer metadata supports keeping in-region data in-region.

What Gatekeeper does **not** do, and what no healthcare organisation should expect from it: it does not de-identify data to a clinical-research standard (Safe Harbor or Expert Determination — those require different toolchains entirely), it does not satisfy the de-identification requirements for HIPAA-compliant secondary use of patient data, and it does not replace the consent management or the access-control model in the underlying clinical-information-system (it operates on AI traffic specifically, not on the full data layer).

## Operational shape that fits a healthcare deployment

The deployment topology that fits a typical healthcare organisation depends on the organisation's existing infrastructure choices. Hospitals running Kubernetes for clinical applications fit the **Kubernetes sidecar topology** documented in the [DEPLOYMENT guide](../../../runtime/docs/DEPLOYMENT.md). Smaller clinics or specialist practices running on more traditional virtualised infrastructure can use the **single-host topology** with APISIX and the sidecar as systemd services on the same host. Both topologies isolate the sidecar to loopback (`127.0.0.1:8081` only, no external network exposure) and provide the same audit-trail durability.

The audit pipeline lands events in PostgreSQL with monthly partitioning and configurable retention (the seven-year default suits most healthcare audit windows; longer retention is a matter of configuration). The append-only design — `DO INSTEAD NOTHING` rules on UPDATE and DELETE inside PostgreSQL itself — makes the audit trail tamper-proof even from a privileged database operator, which matters for healthcare audit defensibility. The audit events themselves never carry the original PII — only the field path that was masked, the strategy that was applied, the rule identifier, and a metadata excerpt — which means the audit trail itself does not become a secondary repository of patient data.

For organisations integrating Gatekeeper with existing clinical-information-system audit infrastructure, the audit-event schema is documented in [`DB_SCHEMA.md`](../../04_design/DB_SCHEMA.md) and the events can be exported to S3, Splunk, or any other downstream audit aggregator through standard PostgreSQL replication or change-data-capture tooling. Gatekeeper does not assume its audit table is the only audit destination.

## Honest about what's not in v0.1.1

Four minor gaps are worth knowing about for healthcare evaluations:

The `ariactl` CLI promised in our original architecture is deferred to v0.2; in v0.1.1 the canary control endpoints are exposed through the APISIX plugin control plane and operators script against them with `curl`. The sidecar's deeper prompt-side security capabilities (`PromptAnalyzer` for vector-similarity injection scoring, `ContentFilter` for output moderation) are stubs in v0.1.1 — these are part of the v0.3 enterprise CISO tier; the v0.1.1 community tier covers prompt-injection through a regex-based scan, which catches known patterns but not novel vector-similar variations. Reversible tokenisation backed by AES-256 is deferred to v0.2 — the v0.1 `tokenize` strategy emits a non-reversible hash, which is sufficient for audit-trail use but not sufficient for tokenisation-and-detokenisation flows where the original value needs to be restored later. Clinical-specific NER (medications, conditions, procedures, lab values) requires loading a clinical-NER model — the engine code in v0.1.1 is community-tier and the architecture supports it, but the bundled multilingual model is general-purpose Turkish-BERT and is not clinical-specific; healthcare organisations needing clinical-NER should plan to either load a clinical model through the existing engine architecture or wait for the v0.3 enterprise DPO tier which will bundle clinical-specific NER variants.

A healthcare organisation evaluating Gatekeeper should expect the v0.1.1 release to satisfy general PII masking, cost control, and progressive-delivery use cases for AI workloads, with the four gaps above documented as known limitations. For deployments where clinical-specific NER is critical, plan for the engine plug-in pattern as the v0.1 path forward; for deployments where reversible tokenisation is critical, v0.2 is the appropriate target.
