# 3e-Aria-Gatekeeper — Architecture Overview

This is the public-facing technical architecture document. It explains how Gatekeeper is put together, what each piece does, and how the pieces interact. It is the document to read after the [datasheet](DATASHEET.md) and before [the internal HLD and LLD](../../03_architecture/HLD.md), which assume more familiarity with our specific architecture-decision-record vocabulary.

The intended reader is a senior engineer evaluating Gatekeeper for a real deployment, or a developer curious about how an open-source AI governance product is actually structured.

## The big picture

Gatekeeper is two pieces of software that ship together and run together. The first piece is **three Lua plugins** that load into Apache APISIX and run inside its OpenResty workers — they are what handles the per-request work on the gateway hot path. The second piece is **`aria-runtime`**, a small Java sidecar that handles the work that does not fit a Lua scripting environment — token counting against an OpenAI-compatible BPE encoding, named-entity recognition for multilingual PII detection, structural diff computation between primary and shadow responses, and the background drain of the audit-event buffer.

The two pieces communicate over **HTTP/JSON over loopback TCP** (`127.0.0.1:8081`). The sidecar binds exclusively to the loopback address and refuses to bind externally; the trust boundary is the Pod (in Kubernetes deployments) or the host (in single-host deployments), and a NetworkPolicy template in the Helm chart further restricts ingress to the same Pod as defence-in-depth.

If you have ever deployed an Envoy sidecar with Istio or a Linkerd proxy, the topology will feel familiar: APISIX is the data-plane process that traffic passes through, and the sidecar is a co-located helper that the data-plane process calls out to for work it cannot or should not do itself. The shape is borrowed from the service-mesh world; the contents are specific to AI governance.

## What runs where

A typical Pod has two containers in it. The **APISIX container** runs APISIX itself with the three Aria plugins loaded; it serves traffic on whatever ports the route configuration says (typically 9080 and 9443 for HTTP and HTTPS). The **`aria-runtime` container** runs the Java sidecar and listens on `127.0.0.1:8081`.

Outside the Pod, two data stores back the system. **Redis** carries all the real-time state: per-consumer token quotas and dollar budgets, circuit-breaker state for both the upstream LLM providers and the sidecar bridges, canary stage configuration, and the audit-event buffer that the sidecar drains. **PostgreSQL** carries the durable audit and billing tables — append-only `audit_events`, `billing_records`, and `masking_audit`, all partitioned monthly with seven-year retention and tamper-proof through database-level rules that prevent UPDATE and DELETE.

Both data stores can be shared across multiple Gatekeeper deployments — a single Redis cluster can serve quota state for several APISIX instances if the consumer key namespacing is consistent, and a single PostgreSQL instance can collect audit events from multiple sidecar instances. The Helm chart does not assume either data store is dedicated to Gatekeeper.

For observability, both halves emit Prometheus metrics on the standard `/metrics` endpoint. Three Grafana dashboards ship in `dashboards/` — one per plugin module — and the Helm chart auto-discovers them via the standard sidecar-injection annotations. A PrometheusRule template is included with starter alerts that operators tune to their own environment.

## The plugin layer in detail

Each of the three Lua plugins is a single file in `apisix/plugins/` plus a handful of shared library modules in `apisix/plugins/lib/`. They follow APISIX's standard plugin pattern — a JSON schema for configuration, lifecycle hooks (`access`, `body_filter`, `header_filter`, `log`) that wire into the APISIX request lifecycle, and metric emission through the standard Prometheus integration.

`aria-shield` runs in the `access` phase to do quota checks, prompt-side scanning, and provider routing; in the `header_filter` phase to start response transformation; in the `body_filter` phase to compute approximate token counts and write the consumed tokens back to Redis; and in the `log` phase to emit audit events into the Redis buffer. Most of the request hot path is in the `access` and `body_filter` phases, where every additional millisecond is felt by the user; the `log` phase work is post-response and cannot affect user-perceived latency.

`aria-mask` runs primarily in the `body_filter` phase (because it operates on the response payload from the upstream) and in the `log` phase (for masking audit events). The two-pass design — JSONPath rules first, then PII auto-detection with optional NER bridge — happens entirely within the `body_filter` phase. The NER bridge call to the sidecar is synchronous from the plugin's perspective: the plugin waits for the sidecar response before completing the mask step. This is acceptable because the alternative (returning the response to the client and then editing it asynchronously) is incompatible with the response actually being modified.

`aria-canary` is the most complex of the three. Its `access` phase implements the consistent-hash routing decision and the shadow-fire logic (when shadowing is enabled). Its `body_filter` phase captures the primary response into a per-request context. Its `log` phase is where the bulk of the work happens — error-rate accounting, stage progression checks, auto-rollback decisions, and (when shadowing was active) the structural-diff request to the sidecar. Putting most of the work in the `log` phase is intentional: nothing the canary plugin does in the `log` phase can affect the response the client receives, which means experimentation is safe by construction.

The shared library modules are worth a brief mention because they encode patterns that the plugins reuse. `aria-core.lua` is Redis pooling, structured logging, Prometheus metric emission with cardinality guards, and the audit-event buffer LPUSH primitive. `aria-pii.lua` is the eight built-in PII regex patterns with checksum validators (PAN by Luhn, Turkish national ID by mod-11, IBAN by ISO-7064, IMEI by Luhn). `aria-quota.lua` is the quota check, dollar-cost calculation against the per-model pricing table, and overage policy resolution. `aria-mask-strategies.lua` is the twelve mask strategy implementations (`last4`, `first2last2`, `hash`, `redact`, `full`, `mask:email`, `mask:phone`, `mask:national_id`, `mask:iban`, `mask:ip`, `mask:dob`, `tokenize`). `aria-provider.lua` is the per-provider request and response transformation logic for OpenAI, Anthropic, Google Gemini, Azure OpenAI, and Ollama. And `aria-circuit-breaker.lua` is the generic per-endpoint circuit breaker (state stored in `ngx.shared.dict` so it survives across OpenResty workers) that the bridge calls use to short-circuit when the sidecar is unhealthy.

## The sidecar in detail

`aria-runtime` is a Spring Boot 3.4 application built on the Java 21 toolchain. Its job is to be the place where the work that does not fit a Lua scripting environment lives. The codebase is small — about 4000 lines of production Java plus a 128-test JUnit suite — and intentionally so.

It exposes three categories of HTTP endpoint. **Bridge endpoints** (`POST /v1/diff`, `POST /v1/mask/detect`) are what the Lua plugins call when they need the sidecar's heavy work. **Operational endpoints** (`/healthz`, `/readyz`, the Spring Actuator's `/actuator/info`, `/actuator/metrics`, `/actuator/prometheus`) are what container orchestrators and monitoring systems call to verify the sidecar is alive and to scrape its telemetry. **Forward-compat gRPC services** (`MaskServiceImpl`, `CanaryServiceImpl`, `ShieldServiceImpl`) exist on the gRPC port for non-Lua callers — other sidecars, admin tools, internal microservices — that may eventually want to invoke the same domain logic without going through the HTTP layer. The cross-transport engine-sharing pattern (a Spring `@Service` bean injected into both the `@RestController` and the `@GrpcService` impl) ensures that the underlying domain logic is the same regardless of how it is called.

Inside the sidecar, three domain services do the heavy work. **`DiffEngine`** computes the structural diff between two HTTP responses — status code comparison, header delta, body-structure similarity. The body-structure comparison is the interesting part: it is not a textual diff (which would be noise-heavy for JSON payloads where field order or insignificant whitespace differs) but a structural comparison that surfaces meaningful differences (added or removed fields, value-type changes, significant value differences) while ignoring incidental ones. **`NerDetectionService`** orchestrates the NER pipeline: it is configured with a list of `NerEngine` implementations (the v0.1 community tier ships two, OpenNLP for English and DJL with ONNX Runtime for multilingual), it calls each engine in order, it unions and dedupes the results, and it filters by minimum confidence. The pluggable engine interface means adding a new language is a matter of writing a new engine class and listing it in `aria.mask.ner.engines` — no change to the bridge contract. **`TokenEncoder`** implements OpenAI-compatible token counting using jtokkit; per-model encoding lookups are cached, and unknown models fall back to `cl100k_base` with a `FALLBACK` accuracy flag so downstream billing pipelines can apply provider-specific correction.

Two background concerns run on a Spring scheduler. **`AuditFlusher`** drains the `aria:audit_buffer` Redis list every five seconds (configurable via `aria.audit.flush-interval-ms`) using LPOP, persists each event into the PostgreSQL `audit_events` table via the R2DBC `PostgresClient`, and exposes `persistedTotal` and `failedTotal` counters for Prometheus alerting. The decision to use LPOP polling rather than an HTTP bridge for audit ingestion (ADR-009 in the architecture decision register) was made deliberately and is worth understanding: the audit pipeline is asynchronous by design — the request must not block on durable persistence — and the Redis buffer is the natural decoupling point between Lua emit and Postgres persist; adding an HTTP hop would either stall the request critical path (if synchronous) or drop events on sidecar restart (if fire-and-forget), neither of which beats the LPOP polling design's natural sidecar-restart safety. **Flyway** runs once at startup, applies the schema migrations (`V001__create_schema_and_enums.sql`, `V002__create_billing_and_masking_tables.sql`, `V003__create_partitions_and_maintenance.sql`), and closes its JDBC connection — runtime queries continue through the asynchronous R2DBC client, no JDBC pool persists.

The persistence layer is split deliberately. **R2DBC for runtime queries** keeps the request-handling path non-blocking — `PostgresClient.insertAuditEvent` and `insertBillingRecord` are async and integrate cleanly with the virtual-thread executor that handles each request. **JDBC for Flyway** because Flyway has no R2DBC equivalent in the ecosystem; the JDBC driver is included in the build solely for migration use, and Flyway closes its connection after applying migrations. Both clients use the same `aria.postgres.*` connection coordinates, so there is one source of truth for the database connection in the configuration.

## How a request flows through

The story is easier to follow as a single example. Imagine a customer support application that calls Gatekeeper with an OpenAI-format chat-completion request. The request arrives at APISIX over HTTPS.

APISIX runs its standard authentication plugin and attaches the consumer identity to the request context — the consumer's role, their consumer ID, their associated metadata. The Shield plugin starts in the `access` phase: it reads the consumer's quota state from Redis (a sub-2ms pipelined GET), runs the regex prompt-injection scan against the prompt content, runs the PII regex scan to detect personal data in the prompt and (depending on the policy) either masks, blocks, or warns, and selects an upstream LLM provider through the routing policy (default provider with circuit-breaker-aware failover to alternates). The request body is transformed into the format the selected provider expects, and the request is forwarded.

When the response comes back from the upstream, the Shield plugin runs the `header_filter` phase to start transforming the response headers back into OpenAI shape, and the `body_filter` phase to do the body-level transformation, the approximate token count, the quota INCRBY against Redis, and (for streaming responses) the chunk-by-chunk forwarding without buffering. By the time the response leaves Shield, it is in OpenAI shape, the quota has been updated, and the headers carry the `X-Aria-*` metadata that operators use for log correlation.

The Mask plugin runs next, in the `body_filter` phase. It reads the consumer's role from the request context, resolves the role to a masking policy (administrators get fewer fields masked than external partners), and applies the JSONPath field rules — for each field that the policy says should be masked, it applies the configured mask strategy from the twelve available. It then runs the PII auto-detector: the regex pass first (which catches structural PII shapes — PAN, Turkish ID, email, IBAN, etc., with checksum validation where applicable), and then optionally the NER bridge for any candidate fields the regex did not classify. The NER bridge call is an HTTP `POST /v1/mask/detect` to the sidecar with the field text and the language hint; the sidecar runs the call through `NerDetectionService`, which fans out to whatever engines are configured (typically OpenNLP for English plus DJL+ONNX for multilingual), unions the results, and returns the entity spans. The Mask plugin merges the regex hits with the NER spans, applies mask strategies to the combined set, and emits a masking-audit event into the Redis buffer (metadata only — never the original values).

The Canary plugin runs in parallel, on routes where canary or shadow is configured. In the `access` phase, it makes the consistent-hash routing decision — if the hash falls below the canary percentage, the request goes to the candidate upstream; otherwise it goes to the baseline. If shadowing is also enabled, the plugin fires a copy of the request to the shadow upstream as fire-and-forget, with an `X-Aria-Shadow: true` header so any downstream Canary instance refuses to shadow it again (recursion guard). The primary response is returned to the client through the normal path.

After the response has been returned to the client, the Canary plugin's `log` phase work runs. If a shadow was fired, the plugin captures both responses from the per-request context and sends them to the sidecar's `POST /v1/diff` endpoint; the sidecar's `DiffEngine` runs the comparison and returns a structured `DiffResult` with status, header, and body-structure deltas. The plugin emits the diff results as Prometheus metrics that operators read from the canary Grafana dashboard. Error-rate accounting runs in the same `log` phase: the plugin updates the canary error-counter in Redis, compares the canary error rate to the baseline using a sliding window, and if the error delta sustains above the threshold for the configured duration, it auto-rolls-back — traffic to the canary drops to zero, a webhook fires, and the canary state goes into `ROLLED_BACK`.

While all this is happening, the audit events that Shield and Mask emitted are sitting in the `aria:audit_buffer` Redis list. On the sidecar's next 5-second tick, `AuditFlusher` LPOPs the events one at a time (up to 100 per tick), parses each, calls `PostgresClient.insertAuditEvent` (asynchronous), and increments `persistedTotal`. The `audit_events` table receives the rows; the database-level append-only rules ensure that nothing — not even a privileged operator — can subsequently UPDATE or DELETE them. The audit trail is durable and tamper-proof from inside the database itself.

## Failure modes and resilience

A governance system that is not honest about how it fails is not a governance system that operators can trust in production. The failure modes worth knowing about are documented in the [LLD](../../04_design/LLD.md) and the operational sections of the [DEPLOYMENT guide](../../../runtime/docs/DEPLOYMENT.md), but the high-level shape is worth describing here.

When the **upstream LLM provider fails** — 5xx, timeout, network error — the Shield plugin's circuit breaker opens after a configurable consecutive-failure threshold and traffic falls over to the next provider in the configured fallback chain. If every provider fails, the client receives a 503 with a structured error code; nothing silently retries forever, and nothing silently swallows failures.

When **Redis fails** — connection timeout, refused connection — the Lua plugins fall back to the configured policy. For quota: fail-open or fail-closed, per-route configuration. For canary: use the last-known state from `ngx.shared.dict` (which survives across worker restarts but not across pod restarts) and pause stage progression. For audit emit: the LPUSH fails, the event is dropped, and a Prometheus counter increments — the request critical path is not blocked.

When the **sidecar's HTTP bridge fails** — connection refused, timeout, 5xx — the Lua plugin's outer circuit breaker (`aria-circuit-breaker.lua`) trips after the configured failure threshold and short-circuits subsequent calls during the cooldown window. The plugin applies the configured fail-mode: for the NER bridge, this is either `open` (return regex-only results) or `closed` (redact all candidate fields); for the diff bridge, it is to skip the diff entirely (the primary response is already in the client's hands by this point, so the only consequence is a missing diff metric).

When **PostgreSQL fails** — `insertAuditEvent` returns an error — the audit event remains in the Redis buffer (since `LPOP` was already executed, the event is in the sidecar's hands at this point, but the sidecar logs the failure, increments `failedTotal`, and drops the event for poison-message containment). Subsequent tick attempts will continue processing the remaining buffered events; the failed event is lost from durable persistence. The v0.3 candidate mitigation is a dead-letter queue (`aria:audit_dead_letter`) for operator-driven replay; the v0.1 design choice was to favour pipeline simplicity over guaranteed delivery for events whose payload had already failed processing.

When **the sidecar itself restarts** — deploy, crash, OOM kill — the Lua plugins continue serving traffic. Bridge calls fail open (NER returns regex-only, diff is skipped). Audit events accumulate in the Redis buffer until the sidecar comes back; on restart, `AuditFlusher`'s next tick begins draining the backlog. The Redis 1-hour TTL on the buffer list bounds the loss window; an outage longer than that drops events older than the TTL.
