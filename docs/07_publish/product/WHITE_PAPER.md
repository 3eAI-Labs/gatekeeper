# AI Gateway Governance: A Reference Architecture

**A 3e-Aria-Gatekeeper white paper · v1.0 · 2026-04-25**

## Abstract

When an organisation begins routing application traffic to large-language-model providers — OpenAI, Anthropic, Google, Azure OpenAI, self-hosted Llama or Qwen variants — three problems show up almost simultaneously, and they are problems that no individual application team is well-positioned to solve. Costs become unpredictable in ways that the existing FinOps practice does not anticipate. Sensitive data leaves the organisation in prompts that no compliance review approved. And every change to a model, a prompt template, or a routing rule is a release that nobody is quite sure how to test in production without breaking real users.

This paper makes the case that the right place to address all three problems is the API gateway layer that already sits in front of those calls — not new instrumentation inside every application, not new infrastructure standing in parallel to what is already deployed. It walks through the design principles behind 3e-Aria-Gatekeeper, an open-core governance suite for Apache APISIX that addresses the cost, privacy, and progressive-delivery problems from a single place. And it documents the trade-offs we made along the way, including the ones that did not work and were reversed.

The intended reader is a platform engineer, security architect, or technical product owner who is responsible for an AI integration that has crossed from "a few experiments" into "a real production load" and is starting to ask the questions that the proof-of-concept phase did not surface.

---

## 1. The shape of the problem

The story is by now familiar enough that it feels almost predictable. An application team integrates an LLM API to ship a useful feature — a customer support assistant, a code-review helper, a document-summary endpoint. The integration works, users like it, more teams adopt the same pattern. Six months later, several things have happened in parallel that nobody planned for as a system.

The monthly bill from the model provider has tripled, and finance is asking which team owns which slice of the cost. The legal team has discovered, through a security review of an unrelated feature, that personal data has been flowing in user prompts to a third-party processor — sometimes obviously (a customer's national ID typed into a support chat), sometimes subtly (a tool call returning a payload that contains an IBAN). The engineering team that owns the deepest LLM integration wants to switch from the model the company started with to a newer, cheaper, faster one — but they have no good way to compare the two on production traffic without risking real users seeing the worse of the two responses. And the security team has read enough about prompt injection to want to know what the organisation is doing about it.

These four conversations — cost control, data privacy, progressive delivery, and prompt-side security — usually start in four different rooms with four different stakeholders. They all share a common architectural feature: they are problems about **inflow and outflow at the boundary** between the application and the model provider, not problems about what happens inside the application or inside the model. Solving them inside individual applications is possible, and many teams have started down that road. But solving them inside applications has the same set of problems that solving anything cross-cutting inside applications always has: it is duplicated work, it drifts between teams, and it is invisible from the platform side, which means that platform engineering, security, and finance can't see the controls actually in effect.

The right architectural location is the gateway. If the organisation already runs an API gateway — and most organisations of meaningful size do — the gateway is already the choke point through which AI traffic flows. Adding governance there means adding it once, in a place that the platform team already operates, with a posture that is uniform across every application that uses the gateway.

The interesting design question is not whether the gateway is the right place. The interesting design question is what governance at the gateway should actually do, what it should *not* do, and how it should be structured so that the four conversations above all get satisfying answers from one place.

## 2. The gateway as the natural location

There are three credible alternatives to gateway-level governance, and it is worth being explicit about why each one is the wrong default before describing the gateway approach.

The first alternative is **inside the application**. Every team that calls an LLM API wraps the call in a middleware layer that does its own quota counting, its own PII scanning, its own circuit breaking, its own logging. This is what most organisations actually do at the proof-of-concept stage, because it is the path of least resistance for a single team. The problem is that it does not survive scale. By the time five or fifteen teams are doing it, no two implementations are quite the same, the quality of each is bounded by the time the team had available, and there is no organisational view of what is being enforced where. Auditing the controls becomes an audit of every application individually.

The second alternative is **inside a custom proxy** — a new HTTP service that the application calls instead of the LLM provider directly, which then forwards. This concentrates the work in one place and is what some larger organisations have built internally. The problem is that it is a new piece of infrastructure: a new deployment, a new monitoring story, a new on-call rotation, a new failure surface. It also tends to become bespoke very quickly, because the team that owns it is the only one who understands what is in it.

The third alternative is **at the model provider level**, through whatever governance features the provider offers — Azure OpenAI's content filter, OpenAI's organisation-level controls, the per-key rate limits all providers expose. These are real features and useful as defence-in-depth, but they are insufficient as the primary control plane for two reasons. First, they are per-provider — an organisation that wants to use multiple providers (which is increasingly common, both for cost arbitrage and for failover) has to configure the same governance separately in every provider console. Second, they are downstream of the data leaving the organisation, which means data privacy controls in particular cannot live there: by the time the provider sees the request, the personal data has already left.

Gateway-level governance has none of these problems. The gateway is already deployed, already monitored, already on-call. The governance configuration is one place, applied consistently regardless of which upstream provider serves a particular request. And data leaving the organisation can be inspected and modified before it crosses the network boundary.

The remaining design question is which gateway. We chose Apache APISIX as the host, and our reasoning is worth stating. APISIX runs on top of OpenResty, which is a mature production-grade extension of Nginx with a Lua scripting layer designed exactly for the kind of per-request work that gateway plugins do. APISIX has a credible plugin model, an Admin API that operators are already comfortable with, and a deployment story that fits both small docker-compose installations and large Kubernetes setups. It is licensed Apache 2.0, which matters for the open-core model we adopted (more on that below). It is in the CNCF landscape and has a real community. None of these are uniquely true of APISIX — Kong, Tyk, Envoy, and others have similar properties — but APISIX was the gateway that already showed up most often in the operator conversations we had during early scoping, and the Lua-on-OpenResty plugin model fit our design instincts particularly well.

## 3. Three concerns, one place

The architecture we settled on splits gateway-level AI governance into three modules that share one underlying runtime. Each module addresses one of the conversations from §1, and each can be enabled independently — an organisation that only wants cost control today does not need to deploy the masking or canary modules.

**The Shield module addresses cost control, multi-provider routing, and prompt-side security.** It enforces token quotas and dollar budgets per consumer, with the consumer identity coming from APISIX's existing authentication plugins; it routes to multiple upstream LLM providers with per-provider request and response transformation so the application sees a uniform OpenAI-shaped surface no matter which provider serves the request; it implements a Redis-backed circuit breaker that fails over between providers; it scans prompts for known PII shapes and runs a regex tier of prompt-injection detection. It is the module that turns "we use OpenAI" into something the SRE team can run as a normal piece of infrastructure.

**The Mask module addresses data privacy on both sides of the conversation.** It applies field-level masking to responses through JSONPath rules, implements role-based policies that distinguish administrators from support agents from external partners, and runs PII auto-detection on prompts and responses through a combination of regex patterns (with proper checksum validation for formats that have one) and named-entity recognition for everything that does not match a structural pattern. The NER engine is pluggable — Apache OpenNLP for English ships in the box, a multilingual ONNX engine with a default Turkish-BERT model handles non-English content, and adding new languages is a matter of writing a new engine class against a small interface.

**The Canary module addresses progressive delivery and the change-safety problem.** It implements multi-stage traffic splitting with consistent hashing for stable client experience, monitors error rates with sliding-window comparison against a baseline upstream, auto-rolls-back when a candidate sustains error rates above a configurable delta, and supports traffic shadowing — fire-and-forget duplication of live traffic to a candidate upstream — with structural diff comparison between the primary and shadow responses. The structural diff is the key feature: it lets operators see *exactly* how the candidate response differs from the baseline before a single live user sees the candidate's output.

These three modules share an underlying runtime — a small Java sidecar that handles the work that does not fit a Lua scripting environment. The sidecar implements real OpenAI-compatible token counting (using jtokkit, the Apache 2.0 Java port of OpenAI's tiktoken), runs the NER inference for the masking pipeline, computes the structural diff for the canary shadow comparison, and drains the audit-event buffer that all three modules write to into the durable PostgreSQL audit table. The split between Lua and Java is not arbitrary: Lua is excellent at the per-request hot path, Java is excellent at the heavy work, and pushing one onto the other would compromise both.

## 4. Why open core was the right model

Three licensing models were on the table when we started. **Pure open source** — Apache 2.0 everything, no proprietary tier — has the advantage of community-friendly framing and the disadvantage that it provides no economic engine for the long-term work that turns a community project into a product organisations are willing to bet their compliance posture on. **Pure proprietary** — closed source with paid licensing — has the advantage of a clear revenue model and the disadvantage that operators who would otherwise adopt the product as a way to satisfy their *own* governance requirements have no way to evaluate, customise, or operate it confidently. **Open core** — a community-tier under a permissive open-source license, with proprietary enterprise tiers layered on top — has the structural advantages of both, at the cost of having to draw the line carefully between what is open and what is gated.

We chose open core, and we drew the line by **buyer persona** rather than by feature flag. This is the consequential choice and it is worth explaining.

Most open-core models gate features. The community version has the basic capabilities; the enterprise version has the advanced ones, the integrations, the dashboards. This is structurally fine but it has a failure mode: the gating decisions accumulate in ways that turn the community version into an obviously crippled demo, which damages the open-source credibility that motivated the open-core choice in the first place.

We instead drew the line by which **kind of buyer** asks for which kind of capability. The community tier — three Lua plugins and the sidecar engine, all Apache 2.0 — gives a complete enough product that a platform engineering team can ship it into production today and meet the operational governance needs of most organisations. The enterprise tiers layer on the deeper, more specialised capabilities that *specific buyers* eventually need: a continuously-curated prompt-injection corpus and content moderation for the security buyer (CISO); bundled multilingual NER models, DLP-style outbound mask, and tamper-proof WORM audit-log packaging for the privacy buyer (DPO); advanced billing reconciliation, per-team chargeback, and cost forecasting for the finance buyer (CFO).

The result is that the community tier is not a crippled demo of the enterprise tier; it is a complete product with its own ceiling. Organisations that hit that ceiling because they have a specific persona-driven need can adopt the enterprise tier that matches their need, without having to take on the full enterprise scope. Organisations that never hit the ceiling — and many won't — get genuine value from the community tier indefinitely.

This choice also affects how we ship. Every iteration that the community tier benefits from is shipped under Apache 2.0 first, regardless of whether it would have been gateable. The Turkish-BERT NER engine, the structural shadow diff, the audit pipeline drain, the Flyway bootstrap — every single one of these landed in the open-source repository before the enterprise tier even existed. The enterprise tiers are not a place to hide features that should have been open; they are a place to host work that is genuinely persona-specific and that requires the kind of long-term investment (corpus curation, model bundling, billing-pipeline integration) that does not fit a community contribution model.

## 5. Reference architecture

The deployment shape that has worked for the operators we have worked with is the **co-located sidecar pattern**: APISIX and the Java sidecar run as containers in the same Pod (in Kubernetes) or as sibling processes on the same host (in single-host deployments), with all communication between them happening over loopback. This pattern keeps the latency between the gateway and the sidecar bounded, keeps the trust boundary aligned with the Pod boundary (no network exposure of the sidecar's HTTP port outside the Pod), and means there is one operational surface to monitor rather than two.

Inside that Pod, the data flow for a typical AI request is straightforward enough to describe in prose. A client sends an OpenAI-format request to APISIX. APISIX runs its standard authentication plugin to attach the consumer identity to the request context. The Shield plugin reads the consumer's quota state from Redis (a sub-2ms pipelined GET), runs the regex prompt-injection scan, transforms the request into the format the selected upstream provider expects, and forwards the request. When the response comes back, the Shield plugin transforms it back into OpenAI shape, computes an approximate token count Lua-side, increments the quota counter in Redis, and emits an audit event by pushing a JSON blob onto a Redis list. The Mask plugin runs on the response — JSONPath field rules first, then PII regex scan, then optionally an HTTP call to the sidecar's NER endpoint for fields that did not match a structural pattern, applying mask strategies and emitting its own audit event. The Canary plugin (on routes that have it enabled) runs the consistent-hash routing decision, optionally fires a shadow request to a candidate upstream, and after the response has gone back to the client, runs the log-phase work that compares the primary and shadow responses through the sidecar's diff endpoint.

The sidecar, on a 5-second background tick, drains the audit-event buffer in Redis and persists each event into the PostgreSQL audit table — append-only, with database-level rules that prevent UPDATE and DELETE so the audit trail is tamper-proof from inside the database itself. The sidecar's other endpoints are exercised on demand: when a Lua plugin needs an exact token count, an NER detection, or a structural diff, the plugin makes an HTTP call to the appropriate sidecar endpoint and processes the result.

There are two pieces of this architecture that are worth dwelling on, because they were design choices that took several iterations to get right.

The first is the transport between the Lua plugins and the sidecar. Our original architecture (ADR-003 in the registry) called for gRPC over Unix Domain Sockets — a low-latency, kernel-level-protected channel that fit the latency-sensitivity of the request hot path. After several iterations of trying to make a Lua gRPC client work robustly inside OpenResty, we reversed the decision (ADR-008) and switched to HTTP/JSON over loopback TCP. The reasons were straightforward in retrospect: Lua does not have a maintained gRPC client library, the alternatives are a custom FFI binding (multi-week effort with ongoing maintenance burden) or no client at all; the latency difference between UDS gRPC (~0.1 ms) and loopback TCP HTTP/JSON (~1–2 ms) is dominated by the LLM provider call (~50 ms to several seconds) by orders of magnitude and is therefore not material; and HTTP/JSON is debuggable with `curl` while gRPC requires `grpcurl` plus the proto file. The cross-transport engine-sharing pattern that emerged from this — a Spring `@Service` injected into both an HTTP `@RestController` and a gRPC `@GrpcService` impl — let us keep the gRPC services in the codebase as forward-compat for non-Lua callers without forcing the Lua hot path through gRPC.

The second is the audit pipeline. Our original FINDING-003 closure plan was to add a third HTTP bridge — `POST /v1/audit/event` — and have the Lua side fire-and-forget audit events through it. An early proposal even considered a hybrid that would write the event both to the HTTP bridge (for fast-path persistence) and to the Redis buffer (for catch-up if the bridge was down). We rejected the hybrid in response to a sharp question from the project owner — *"why two paths?"* — that crystallised the design problem. The two-path design did not actually buy any additional safety: the Redis buffer with TTL is already the natural source-of-truth between Lua emit and Postgres persist, and adding a synchronous HTTP path would either stall the request critical path on sidecar latency (if synchronous) or drop events on sidecar restart (if fire-and-forget). The simpler design — Lua pushes to Redis, sidecar polls Redis with LPOP on a scheduled tick — is what shipped (ADR-009), and it is genuinely better: sidecar restart safety is free (events sit in Redis with TTL until the next tick consumes them), the Lua side stays unchanged, and the failure modes are bounded by Redis behaviour rather than spread across two protocols.

These two ADRs — 008 and 009 — illustrate something we believe deeply about gateway governance design: **the boring choice is usually the right choice**. UDS gRPC felt sophisticated; HTTP loopback was boring; HTTP loopback was right. A two-path hybrid felt clever; LPOP polling was boring; LPOP polling was right. The instinct to reach for the more sophisticated option is, in our experience, almost always wrong in production infrastructure where the operator's debugging tools and the platform team's mental model determine how survivable the system is in a 3am incident. We optimise for that survivability, not for the elegance of the design slide.

## 6. Trade-offs we made on purpose

A reference architecture that does not document the things it gave up is incomplete. The choices below are explicit trade-offs — places where we picked something we believe is right for the v0.1 release with the understanding that we may revisit the choice as the product matures.

**No queue between the gateway and the sidecar.** We considered Kafka or NATS as a buffer between the Lua plugins and the sidecar's heavy work — diff computation, NER inference, audit persistence. We chose Redis as the only queue because the audit pipeline is the only async path that needs queuing semantics, the diff and NER paths are synchronous request-reply by design, and adding a second message broker would compound the operational footprint with no proportional benefit at the v0.1 scale. ADR-006 documents the reasoning. We expect this to be the right call through the v0.x line; if we hit operators who are running gateway traffic at a scale where Redis becomes a bottleneck for the audit drain, we will revisit.

**Sidecar is JVM, not Go or Rust.** The performance argument for a Go or Rust sidecar — especially around NER inference — is real. We chose Java 21 because the JVM has the most mature ecosystem for the specific work the sidecar does (jtokkit for tokenisation, OpenNLP for English NER, DJL+ONNX Runtime for multilingual NER, R2DBC for async PostgreSQL, Lettuce for async Redis), because virtual threads remove most of the historical concurrency-overhead argument against the JVM, and because the platform engineers we have talked to are more often comfortable operating a JVM service than a Go or Rust one. We expect this to be the right call for the foreseeable future; an operator who needs sub-millisecond NER inference at high throughput could justify a Rust or Go alternative engine, and the `NerEngine` interface is small enough that one could be written.

**No managed service.** We do not run Gatekeeper as a hosted service. Operators run it themselves, in their own infrastructure, with their own data stores. This is a deliberate choice — the entire point of governance at the gateway is that the gateway is *in the operator's trust boundary*. A hosted version of Gatekeeper would put the data we are trying to protect from third-party LLM providers into the hands of yet another third-party processor, which is precisely the opposite of what most of the operators we have worked with want. We do not rule out a managed offering forever, but for v0.1 the deployment shape is exclusively self-hosted, and the documentation is written accordingly.

**No Prompt-as-a-Product layer.** Some governance products in the AI space conflate gateway governance with prompt management — versioning prompts, A/B testing prompt templates, providing a UI for non-technical users to edit prompts. We deliberately do not do this. The Canary module supports A/B testing at the routing level (route a percentage of traffic to a different upstream that may have a different prompt) but there is no in-product prompt editor, no prompt versioning, no template store. The operators we have worked with already have prompt management solved (often through their existing CI/CD or feature-flag tooling), and adding a competing prompt store would complicate the product without addressing a real gap.

**Compliance framing as capability statement, not certification.** Throughout the documentation — including this paper — we describe what the product *does* in support of operator audits in regulatory contexts (KVKK, GDPR, PDPL, PCI-DSS scope hygiene). We do *not* claim the product certifies compliance with any framework, and we explicitly disclaim PCI-DSS compliance because that would require an audited cardholder-data environment which remains the operator's audit boundary. This framing matters: certification claims that would not survive a regulator's first question erode the credibility of the rest of the documentation. The capability-statement framing is honest, it is durable, and it gives operators the language to describe what the product does to their own auditors without implying things that would have to be retracted.

## 7. What we learned shipping it

The v0.1 release of Gatekeeper had a complicated path to shipping, and the lessons from that path are part of the documentation we publish. Most product white papers do not talk about what went wrong; we are going to, because we think the lessons matter for any team building governance infrastructure of their own.

**Spec drift is the default outcome of small teams shipping fast.** The original v0.1 baseline was tagged on 2026-04-08 with a high-level design, a low-level design, an error-code registry, and an architecture decision register. Over the seventeen days between 2026-04-08 and 2026-04-25, the running code drifted from the spec — feature work shipped (the NER bridge, the structural shadow diff, the real token counter, the HTTP transport replacing the originally-planned UDS gRPC) and the spec did not catch up. We caught the drift because we ran an adversarial spec review that compared the running code to the documentation systematically; that review surfaced fifteen findings, six of them critical, including the audit pipeline that had been documented as working but in fact had no consumer for its events.

The lesson is not "be more disciplined about updating the spec" — that is true but trivial and does not survive contact with shipping pressure. The lesson is **make spec drift cheap to detect and mandatory to close**. The mechanism we put in place — a `GUIDELINES_MANIFEST.yaml` with a `phase_gates.require_human_signature` rule that makes silent acceptance of Phase 6 artefacts a process violation, plus a `/phase-review` skill that runs the adversarial review as a fresh-context audit — turns the cost of detecting drift from "depends on a human catching it in code review" to "automated and routine". This is not a fix for the underlying tendency of fast-moving teams to outpace their documentation; it is an acknowledgment that the tendency exists and a control that catches it before it becomes a credibility-damaging surprise.

**The audit pipeline is the most under-attended part of every governance system we have seen, including the one we built.** Through the v1.0 baseline, the audit pipeline in our own product was incomplete — the Lua side correctly pushed events to Redis, but the sidecar consumer did not exist. This was documented as a "Must-priority" business rule (BR-SH-015) and was assumed to be working until the adversarial review surfaced that no caller of `PostgresClient.insertAuditEvent` existed in the entire codebase. We closed the gap (ADR-009 documents the chosen design) within hours of the finding, but it stayed broken for seventeen days because nobody had explicitly tested that audit events actually landed in the audit table. The lesson: audit pipelines deserve integration tests that verify end-to-end persistence, not unit tests that verify the Lua side emits and unit tests that verify the Java side persists. The two are separate things and the gap between them is exactly where audit pipelines silently fail.

**Compliance framing is a discipline, not a one-time decision.** Our v1.0 release notes claimed PCI-DSS compliance in language that, on re-reading, would not have survived a regulator's first question. We reframed it (Karar 3 in the project's decision history) as "scope hygiene" — capability statements about what the product does to support operator audits, with explicit disclaimers about what it does not do. This was uncomfortable to write because it sounds less impressive than the original framing, but it is honest, and the long-term cost of an over-claim that has to be retracted is much higher than the short-term marketing cost of the more careful language. The discipline we adopted — every framework reference is a capability statement, the `feedback_compliance_framing` memory locks it for future iterations — is something we recommend to anyone shipping governance tooling.

**Operator documentation is a feature, not a chore.** The same week we shipped the audit pipeline closure and the Flyway bootstrap, we rewrote the operator documentation end-to-end — the QUICK_START became a ten-minute governed-call walkthrough, the CONFIGURATION reference became an operator source-of-truth with every knob explained, the DEPLOYMENT guide documented three deployment shapes (docker-compose, single-host, Kubernetes sidecar) with their respective trade-offs. This was as much engineering work as the audit-pipeline closure itself, and it was the work that determined whether operators we hand the product to actually succeed in deploying it. Treating documentation as a deliverable on the same level as code, with the same review and quality bar, is something we picked up from the spec-drift episode and have kept since.

## 8. Closing

The point of all of this is that AI gateway governance is a real category — not a future category, a present one. Organisations that route any meaningful traffic to LLM providers face cost, privacy, and change-safety problems whether they have addressed them yet or not. The set of architectural choices for addressing those problems converges on the gateway layer for reasons that are structural rather than incidental, and the implementation choices within that converge on a small number of patterns: separation of the per-request hot path from the heavy processing, async persistence with bounded buffers, layered circuit breakers, capability-statement-only compliance framing, and an honest acknowledgment of what the v1.0 product does and does not do.

3e-Aria-Gatekeeper is one implementation of those patterns. The community tier is on GitHub under Apache 2.0; the architecture decision records, the code, and the spec are all readable. If you are building gateway governance of your own — at your own organisation, or as a competing product — we hope this paper was useful as documentation of one design path through the problem space. If you are evaluating the product itself, the [QUICK_START](../../05_user/QUICK_START.md), the [DATASHEET](DATASHEET.md), and the [release notes](../../06_review/RELEASE_NOTES_v0.1.0_2026-04-25.md) (which honestly enumerate the v0.1 known gaps) are the right next reads.

We would rather ship a smaller product that an operator can run with confidence than a larger product that an operator has to take on faith. Everything in this paper, and everything in the code, is built on that preference.
